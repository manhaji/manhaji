4 integration tiers · cheapest → richest
Tier 0 · today's plan

Manhaj inbox only

What: teacher logs into Manhaj to read + reply. No Outlook integration.
Risk: friction — teachers won't bother. Threads die.
  • Parent sees: Manhaj inbox ✓
  • Teacher sees: Manhaj inbox only ✗
  • Audit log: clean ✓
Build effort~1d
Teacher adoptionLow
Tier 1 · recommended for pilot

Outlook BCC + reply-to-Manhaj

★ START HERE
What: Manhaj sends parent emails via Resend. Every message BCCs the teacher's school email so it lands in Outlook automatically. Replies have a unique reply+thread-abc123@manhaj.app Reply-To address; Resend webhook captures the reply and posts it back to the thread.
  • Parent sees: Manhaj inbox ✓
  • Teacher sees: Outlook inbox ✓ (with "View in Manhaj" deeplink)
  • Teacher replies from Outlook → captured by Manhaj ✓
  • Audit log: clean ✓
Build effort~3d
Teacher adoptionHigh · no training
Tier 2 · school #2 onwards

Shared mailbox forwarder

What: school IT creates parents@iso.edu.om shared mailbox. Manhaj forwards inbound parent replies to it; teachers reply from this address; replies route back via SMTP.
  • Looks more professional from the school's brand
  • Easier for schools that already use shared mailboxes
  • Requires IT setup per school
Build effort~1w
Teacher adoptionHigh
Tier 3 · scale phase

Microsoft Graph API

What: deep Microsoft 365 integration. Teacher signs in once with their school O365 account; Manhaj reads/writes their Outlook folder. Threads are real Outlook items — bidirectional sync.
  • Best teacher experience — zero context switch
  • Optional Manhaj UI for parents only
  • Needs Microsoft App Registration + per-school admin consent
  • Security review heavier
Build effort~3-4w
Teacher adoptionExcellent

Open questions worth discussing
1. Is BCC enough for pilot, or do we need shared-mailbox from day one?
BCC = teacher gets a personal copy. Shared mailbox = the school's parents@iso.edu.om address is the brand. ISO likely has a 365 tenant already — worth one IT conversation before we lock the design.
2. Should the teacher's Outlook reply skip Manhaj entirely if they want?
Risk: the audit log misses the message. Pro: zero friction. Compromise: Manhaj always captures (via reply-to routing) but teachers don't have to log into Manhaj — they can do everything from Outlook.
3. Behaviour-alerts + admin-broadcasts: send AS the teacher or AS the school?
If "as the teacher" then their personal email is the From — feels personal but breaks if they leave. If "as the school" then noreply@iso.edu.om sends — cleaner but cold. Probably depends on template.
4. Read-receipts / opens — do teachers see "parent opened your message"?
Manhaj can capture this. Some schools want it (so they know if the parent saw the behavioural alert), some find it invasive. School-configurable.
5. Microsoft Graph integration · when?
Probably triggered by the 3rd or 4th school deal. ISO pilot starts on Tier 1 (BCC) — it's the cheapest path to "looks great, no training required". Graph API replaces the BCC plumbing later without affecting parents at all.

My recommendation: ship Tier 1 (Outlook BCC + reply-to-Manhaj) in the IA build. Teachers don't have to learn anything new; parents get a clean threaded inbox; everything is audited. We re-evaluate Tier 2/3 after the ISO pilot when we know which friction points actually matter.
